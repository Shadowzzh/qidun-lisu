## 1. 设计目标

本设计文档定义 `lisu-official-site` 首页第二阶段方向：将首页高保真复刻到 `ai-website-cloner-template` 首页的主骨架、节奏和站点壳，同时把内容主体替换为“北京骊甦科技企业级私有化 AI 知识智能平台方案”。

本次设计目标不是重新设计一个新首页，而是把当前“8 个并列 section 的内容首页”收敛回模板首页的视觉骨架中，让页面既符合骊甦内容文档，也符合用户对模板复刻强度的要求。

## 2. 已确认决策

以下决策已经在设计讨论中确认：

- 复刻强度采用“高保真复刻”。
- 首页主内容骨架应尽量贴近 `ai-website-cloner-template` 首页。
- `Header` 和 `Footer` 也应尽量贴模板的视觉节奏，但内容改成骊甦版本。
- 首页图片优先只使用 `archive/ppt-source/images/` 的现有素材。
- 允许对现有素材做裁切、拼接和重组。
- 若缺少合适图片，不补新图，改为带说明文字的占位 icon，明确此处适合的图片类型。
- 首页外层不再维持当前 8 个独立大 section 的视觉分段。
- 首页内容职责仍需覆盖文档中定义的首页 9 类内容，但允许合并进模板的 4 个主区块。
- UI 实现阶段强制遵守 `baseline-ui` 规范。
- 首页完成后，使用 `Playwright MCP` 做主流程验证，使用 `Chrome DevTools MCP` 做结构、运行时和可访问性检查。

## 3. 设计边界

### 3.1 本次范围

- 复刻首页主骨架、主节奏和站点壳。
- 调整首页内容结构，使其映射到模板首页的视觉区块。
- 建立首页运行时图片资源链路。
- 补齐缺图时的说明型占位图方案。
- 为后续实现提供清晰的模块、内容和测试约束。

### 3.2 非目标

- 本次不交付 `/solution`、`/cases`、`/company` 等子页面。
- 本次不复刻模板复杂下拉导航。
- 本次不虚构二维码、备案号、城市信息、新闻体系或联系方式。
- 本次不新增 `archive/ppt-source` 之外的业务内容来源。
- 本次不引入新的视觉主题或重新发明首页布局语言。

## 4. 设计原则

- 模板骨架优先，内容职责内嵌。
- 页面结构优先稳定，文案和素材通过数据层驱动。
- 首页应更像正式官网，而不是按 PPT 页序展开的长文页面。
- `archive/ppt-source` 作为唯一内容归档来源，但不直接作为运行时资源目录。
- 首页缺图时要有稳定降级方案，不能出现坏图或空图片区。
- 能通过替换文案、图片和卡片完成适配时，不重做布局。

## 5. 首页骨架设计

首页外层骨架严格压回模板首页的 4 段主节奏，不再保留当前 8 个并列大 section 的外层结构。

### 5.1 Hero 主视觉区

承载首页“首屏主视觉”职责，内容来源以 `slide-01` 为主。

区块内容：

- 平台标题
- 一句副标题
- 三个核心卖点
- 一张主视觉图

设计要求：

- 保持模板 `Hero` 的视觉角色和首屏占比。
- 首屏必须具备明确视觉主图或说明型占位图。
- 不在首屏加入多余 CTA、语言切换或伪功能入口。

### 5.2 平台总览主区

这一段复刻模板“平台总览 + 大图 + 三卡片”的主骨架。

承载 4 类内容职责：

- 时代之需
- 战略判断
- 方案主张
- 平台总览

设计结构：

- 上半区：总标题与摘要
- 中间：一张强信息主图
- 下半区：3 张摘要卡片

卡片职责建议：

- 卡片 1：时代之需
- 卡片 2：战略判断
- 卡片 3：方案主张

平台总览内容通过总标题、摘要和主图共同承载，不单独拆成新的大 section。

### 5.3 两段 Feature 图文区

这两段沿用模板“图文配图”的节奏，不额外拆分出多个并列板块。

第一段承载：

- 核心能力入口

第二段承载：

- 业务场景价值

设计要求：

- 两段都保持模板图文比例与节奏感。
- 可按模板交错图文方向布局。
- 每段各自只承载一个主主题，避免把多个次级主题塞进同一区块。

### 5.4 底部收束区

底部不再做模板原有“最新动态”，改为“案例与团队摘要 + 价值承诺收束”。

推荐结构：

- 上下文收束文案：价值承诺
- 三张卡片：
  - 案例 1
  - 案例 2
  - 核心团队

设计要求：

- 保留模板底部卡片收束感。
- 不伪造新闻列表或新闻详情链接。
- 案例卡片只做摘要入口感，不展开完整案例细节。

## 6. 内容职责与骨架映射

首页仍需覆盖骊甦首页文档定义的 9 类内容职责，但这些职责不再一一对应独立视觉 section。

映射规则如下：

- 首屏主视觉 → Hero 主视觉区
- 时代之需 → 平台总览主区卡片 1
- 战略判断 → 平台总览主区卡片 2
- 方案主张 → 平台总览主区卡片 3
- 平台总览 → 平台总览主区总标题、摘要和主图
- 核心能力入口 → Feature 图文区第 1 段
- 业务场景价值 → Feature 图文区第 2 段
- 案例与团队摘要 → 底部三卡片区
- 价值承诺与收束 → 底部收束文案

这样处理后，视觉节奏贴模板，内容职责仍然完整。

## 7. 图片与占位策略

### 7.1 总原则

- 所有正式图片优先来自 `archive/ppt-source/images/`。
- 允许对原图做裁切、拼接和重组。
- 不允许在运行时代码中直接引用 `archive` 路径。
- 实际采用的图片应进入 `src/assets/home/*` 运行时资源目录。
- 没有合适图片时，使用说明型占位 icon。

### 7.2 图位分类

首页图片位分为 5 类：

- Hero 主视觉图
- 平台总览主图区大图
- 核心能力区配图
- 业务场景区配图
- 底部卡片缩略图

### 7.3 各图位建议

Hero 主视觉图建议类型：

- 平台总览感主图
- 架构概览主图
- 品牌主 KV

平台总览主图区大图建议类型：

- 七层架构图
- 平台全景图
- 安全与 OVTP 纵向切面图

核心能力区配图建议类型：

- 能力矩阵图
- 工作台与平台能力总览图
- 员工 AI 工作台示意图

业务场景区配图建议类型：

- 业务价值图
- 业务流程图
- 工作台使用场景图

底部卡片缩略图建议类型：

- 案例示意图
- 产品知识图谱相关图
- 团队背书图

### 7.4 占位图规则

说明型占位 icon 必须满足以下条件：

- 使用统一风格的线性 icon 与中性底色。
- 保留与正式图片位完全一致的尺寸比例。
- 占位内容必须说明“建议放什么图”。
- 占位只用于缺图位置，不主动替代已有真实素材。
- 占位风格不能喧宾夺主，不能破坏模板节奏。

占位文案示例：

- `建议放：首页主视觉图`
- `类型：平台总览 / 架构概览 / 品牌主 KV`
- `建议放：七层架构总览图`
- `类型：分层架构 / 平台全景 / 安全纵向切面`

## 8. Header 设计

`Header` 采用“视觉高保真、交互简化”的策略。

保留内容：

- 模板式品牌位
- 模板式高度、留白和水平节奏
- 桌面端导航横排节奏
- 移动端简化菜单

不保留内容：

- 模板复杂下拉菜单
- 语言切换
- 伪造的外部导航体系

导航规则：

- 顶栏只放可用锚点
- 锚点数量控制在 5 到 6 个
- 更细粒度内容层次保留在页面正文内，不强行全部塞入顶栏

推荐顶栏项：

- 首页概览
- 平台总览
- 核心能力
- 场景价值
- 案例团队
- 价值承诺

品牌位规则：

- 若没有可直接使用的 logo，则先使用文本字标“北京骊甦科技”
- 文本字标应占据接近模板 logo 区的空间，不应显得像普通正文链接

## 9. Footer 设计

`Footer` 采用“版式贴模板、内容只用真实资料”的策略。

整体结构保留：

- 上半区多列信息
- 下半区品牌收束

明确不做：

- 假二维码
- 假备案
- 假城市列表
- 假联系方式

推荐上半区分组：

- 网站导览
- 方案能力
- 页面规划
- 品牌说明块

内容规则：

- 有真实可点击页面的内容可以是链接
- 仅为未来规划的内容可先作为普通文本展示
- 品牌说明块承担骊甦品牌、方案定位和一句话摘要

下半区保留：

- 品牌字标或文本品牌名
- 一句站点描述
- `Copyright © 2026 北京骊甦科技`

## 10. 数据与组件边界

### 10.1 运行时组织

- `src/app/page.tsx` 继续保持薄入口，只负责挂载首页。
- 首页总装应尽量贴模板首页结构。
- 首页主组件建议迁移到 `src/components/pages/home/` 组织。
- 首页应拆成 4 个模板主区块组件，而不是维持 8 个并列大 section 组件。
- `Header` 和 `Footer` 继续保留在 `src/components/site/`。
- `src/content/home.ts` 改为按“4 个模板主区块 + 内部锚点内容”组织。
- `src/content/navigation.ts` 只维护顶栏可见锚点。
- 正式采用的图片与占位图进入 `src/assets/home/*`。

### 10.2 数据驱动规则

- 不允许在组件里写死长文案。
- 不允许在组件里直接引用 `archive` 路径。
- 有真实素材时渲染正式图片。
- 无真实素材时渲染说明型占位图。
- 不允许出现 broken image 或空图片区。

## 11. UI 约束

实现阶段强制遵守 `baseline-ui` 规范，重点约束如下：

- 尽量沿用模板已有视觉语言，不主动引入新的主题风格。
- 允许保留模板中已存在的渐变、阴影和图片区块表达，但不额外发散新的视觉特效。
- 标题使用 `text-balance`，正文使用 `text-pretty`。
- 需要类名条件拼接时使用 `cn`。
- 第一版不主动新增动画。
- 不堆叠无必要的渐变、发光、大面积毛玻璃和任意层级。
- 不手搓复杂键盘与焦点行为。

本次目标不是“重新设计 UI”，而是“在 UI 基线约束下做模板高保真内容复刻”。

## 12. 验收标准

首页验收以以下标准为准：

- 外层骨架与模板首页主节奏一致。
- `Header` 与 `Footer` 的视觉秩序接近模板，但不包含假内容。
- 关键图片区位全部成立，有真图用真图，缺图用说明型占位图。
- 首页内容职责仍完整覆盖骊甦首页文档的核心内容。
- 运行时资源链路成立，图片不直接依赖 `archive` 路径。
- 桌面与移动端布局都可正常使用。
- 锚点导航可用。
- 页面更接近企业官网，而不是 PPT 分段堆叠页。

## 13. 测试与验证策略

### 13.1 静态检查

实现完成后应先通过：

- `lint`
- `typecheck`
- 必要的组件测试

### 13.2 Playwright MCP 验证

使用 `Playwright MCP` 完成首页主流程验证，至少覆盖：

- 首页可正常打开
- `Header`、`Hero`、平台总览区、两段 feature 区、底部收束区均存在
- 顶部锚点点击可滚动到对应内容
- 桌面和移动端布局不出现明显断裂
- 缺图位的说明型占位图能正常显示

### 13.3 Chrome DevTools MCP 验证

使用 `Chrome DevTools MCP` 完成运行时检查，至少覆盖：

- 页面 DOM 结构符合预期骨架
- 图片请求无明显 404
- 控制台无明显报错
- 可访问性与最佳实践做一轮基础检查
- 必要时检查关键图片区位的最终渲染结果

### 13.4 验证顺序

推荐执行顺序：

1. 先跑静态检查
2. 再用 `Playwright MCP` 跑首页主流程
3. 最后用 `Chrome DevTools MCP` 做运行时与结构检查

## 14. 主要风险

### 风险 1：只改文案，不改骨架

结果：

- 首页依然会停留在“内容正确，但不够像模板”的状态

应对：

- 以模板 4 段主骨架为实现主线

### 风险 2：把 9 类内容职责错误理解为 9 个视觉大区块

结果：

- 首页重新退回到碎片化 section 排列

应对：

- 明确“内容职责完整”和“视觉骨架贴模板”是两套维度

### 风险 3：图片链路继续停留在声明层

结果：

- `content` 里有视觉字段，但页面无实际图片

应对：

- 实现时必须建立正式图片或占位图的渲染链路

### 风险 4：为贴模板而引入假信息

结果：

- 页面会出现不真实内容，偏离需求

应对：

- 版式可复刻，信息必须真实或明确是非链接规划文本

## 15. 结论

推荐方案为：

- 外层高保真复刻模板首页的主骨架与站点壳
- 内层按骊甦首页文档的内容职责做重新映射
- 图片只用 `archive` 现有素材，缺图时用说明型占位 icon
- UI 实现遵守 `baseline-ui`
- 完成后用 `Playwright MCP` 与 `Chrome DevTools MCP` 做实机验证

该方案能同时满足以下目标：

- 明显提升首页与模板首页的结构相似度
- 保持骊甦内容职责完整
- 避免假内容和过度再设计
- 为后续多页面官网扩展保留稳定边界

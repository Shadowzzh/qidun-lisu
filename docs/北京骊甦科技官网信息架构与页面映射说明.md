# 北京骊甦科技官网信息架构与页面映射说明

Date: 2026-04-29
Status: Draft
Related Requirement: `docs/北京骊甦科技官网需求说明.md`
Project: `ai-website-cloner-template`

## 文档目的

本文件用于在“北京骊甦科技官网建设需求说明”的基础上，进一步明确官网的信息架构、导航结构、页面主题和内容映射关系。

这份文档回答的问题不是“怎么开发”，而是：

- 网站应该由哪些页面组成
- 每个页面各自表达什么主题
- `public/ppt-source` 中的内容应该落到哪些页面
- 哪些路由天然匹配，哪些路由需要做语义转译

## 输入前提

- 网站工程目录：`ai-website-cloner-template`
- 唯一内容来源：`public/ppt-source`
- 品牌主体：北京骊甦科技
- 网站形态：多页面官网
- 结构约束：沿用目标站的多页面层级，但路由命名改为新的业务语义路径

## 官网整体定位

这套官网应被定义为“北京骊甦科技企业级私有化 AI 知识智能平台官网”。

网站整体表达应围绕以下四个核心主题展开：

- 为什么企业现在需要建设私有化 AI 平台
- 北京骊甦科技提供的方案是什么
- 平台能力如何分层展开
- 方案如何在案例、场景和团队层面建立可信度

## 导航结构建议

在保留原有站点层级感的前提下，建议将网站结构整理为以下形式：

```text
首页 -> /

解决方案
主方案总览 -> /solution
员工AI工作台 -> /workspace
算力底座 -> /infrastructure
知识语义层 -> /semantic-layer
AI数据平台 -> /data-platform
安全管控 -> /security

应用场景 -> /scenarios
行业洞察 -> /insights
案例中心 -> /cases
价值展望 -> /value
关于我们 -> /company
```

## 页面映射总原则

- 首页负责建立品牌认知、方案认知和页面分发，不承载全部细节。
- 解决方案相关页面负责拆解平台能力，而不是重复首页文案。
- 案例页负责建立可信度，优先呈现已存在的案例材料。
- 公司页负责品牌收束和团队背书。
- 对于内容匹配较弱的页面主题，应优先保证表达成立，再确定最合适的路由承载。
- 如需兼容历史演示站路径，应通过重定向单独处理，不保留重复内容页。

## 内容支撑强弱评估

### 强支撑页面

- 首页
- 主方案总览页
- 算力底座页
- 知识语义层页
- AI数据平台页
- 安全管控页
- 案例中心页
- 关于我们页

这些页面在 PPT 中均有比较直接的章节支撑，页面主题与内容关系自然。

### 中等支撑页面

- 员工AI工作台页
- 应用场景页
- 行业洞察页

这些页面能够从现有 PPT 中获得足够内容，但需要通过多页内容整合来形成完整网页结构。

### 弱支撑页面

- 价值展望页

这个页面没有天然对应的独立章节，需要由“私有化价值 + 风险与成功因素 + 三年展望”组合形成。

## 页面间分工关系

- 首页负责总览与导流。
- 主方案总览页负责方案全貌。
- 五个解决方案子页负责能力拆解。
- 应用场景页负责讲“怎么用”。
- 行业洞察页负责讲“为什么现在要做”。
- 案例中心页负责讲“已经做过什么”。
- 价值展望页负责讲“做成后会带来什么，以及如何稳妥推进”。
- 关于我们页负责讲“是谁在做这件事”。

## 需要特别注意的内容空缺

- 当前资料中没有适合官网持续更新的新闻体系。
- 当前资料中没有标准意义上的投资者关系内容。
- 当前资料中没有完整的联系方式、地址、表单字段等对外信息。
- 当前资料虽然包含案例与团队素材，但不同页面的图文丰富度不完全均衡。

因此，后续页面设计应优先围绕“方案官网”展开，而不是按资讯型官网或上市公司官网去组织。

## 后续文档建议

在本文件基础上，下一份文档建议补“页面模块拆解说明”，继续明确：

- 每个页面由哪些模块组成
- 哪些模块可以跨页面复用
- 哪些图片和图表优先放在哪些模块
- 哪些内容只适合摘要展示，哪些内容适合完整展开
